Nextcloud Home Server

A self-hosted personal cloud running on an Inovato Quadra ARM SBC — accessible from anywhere over HTTPS via Cloudflare Tunnel, with no open ports on the home router. Containerized with Docker, managed through Portainer, and monitored with Uptime Kuma.

Setup Process
01
OS to eMMC
Flashed fresh Armbian to microSD, booted, then migrated OS to internal eMMC via armbian-config. MicroSD removed after — Quadra boots entirely from eMMC with faster I/O and no card wear.
02
External Storage
USB SSD partitioned with fdisk, formatted ext4, mounted at /mnt/NextcloudStorage, and made persistent via /etc/fstab. Permissions set to www-data for Nextcloud write access. Keeps data off eMMC and separates OS from storage.
03
Docker + Portainer
Docker installed via official install script. Portainer CE deployed as a container with socket access — provides a web UI for managing all subsequent containers without SSH for routine tasks.
04
Nextcloud + MariaDB Stack
Deployed as a Portainer stack via Docker Compose. Nextcloud data volume mapped directly to the external SSD. MariaDB container name used as hostname — Docker internal DNS resolves it automatically. Database credentials kept out of version control.
05
Cloudflare Zero Trust Tunnel
Tunnel created in Cloudflare Zero Trust dashboard, cloudflared deployed as a Docker container with restart policy Always. Public hostname configured to route yoursubdomain.domain.com to localhost:9321. Happy Eyeballs and Chunk Encoding disabled to prevent upload stalls through the proxy.
06
Nextcloud Configuration
Trusted domain added to config.php. overwriteprotocol → https set to force correct link generation behind the reverse proxy. Upload limits raised to 16GB via .htaccess. WebDAV/CardDAV/CalDAV redirects configured in Apache. APCu memory cache enabled.
07
Cron + Uptime Kuma
Background jobs switched to system cron (more reliable than AJAX). Cron job runs cron.php every 5 minutes as www-data. Uptime Kuma deployed as a stack and configured to monitor the cron endpoint at 300-second intervals with heartbeat alerts.
Key Takeaways
Cloudflare Tunnel over port forwarding
Running the tunnel as outbound-only means the router never has an exposed port. The server is accessible from anywhere over HTTPS without touching NAT or firewall rules — and Cloudflare handles DDoS protection at the edge.
overwriteprotocol is non-optional behind a proxy
Without this config flag, Nextcloud generates internal http:// links even when the external connection is HTTPS. Sharing links, app sync, and WebDAV all break silently. This was the single most confusing part of the setup.
Separate OS and data volumes
Keeping the Nextcloud data directory on an external SSD rather than the eMMC prevents OS storage from filling up, extends eMMC lifespan, and makes it easy to swap or expand storage without touching the OS or container configuration.
File locking failures need SQL intervention
Nextcloud's file locking table can deadlock under certain conditions — no UI escape. The fix is to enable maintenance mode, connect directly to MariaDB, truncate oc_file_locks, and restart the container. Knowing this in advance saves significant debugging time.
Portainer makes container ops practical
Managing multiple containers (Nextcloud, MariaDB, cloudflared, Uptime Kuma) via CLI on a headless ARM board gets unwieldy quickly. Portainer's stack-based deployment and web console make restarts, log inspection, and config changes accessible from any browser.
ARM hardware runs this stack reliably
The Quadra handles Nextcloud, MariaDB, cloudflared, and Uptime Kuma simultaneously at low power. Migrating to eMMC and using a dedicated data SSD keeps I/O fast enough that the ARM bottleneck is effectively invisible for personal cloud use.
